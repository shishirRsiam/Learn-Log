
> **“We were unable to decode this CSR. It may be corrupt or in an incorrect format.”**

This means **Name.com’s parser cannot read what you pasted**. That only happens when the CSR is **not a valid PEM CSR**.

---

## The 5 REAL causes (ranked by probability)

### 1️⃣ You did **not paste the FULL CSR**

This is the #1 cause.

A valid CSR **must** look exactly like this:

```
-----BEGIN CERTIFICATE REQUEST-----
MIICzTCCAbUCAQAwgYUxCzAJBgNVBAYTAkJE
...
...
-----END CERTIFICATE REQUEST-----
```

❌ If **any** of these are missing:

* BEGIN line
* END line
* middle base64 block
  → decode fails

---

### 2️⃣ You pasted the **PRIVATE KEY** instead of CSR (very common)

If your pasted text starts with:

```
-----BEGIN PRIVATE KEY-----
```

or

```
-----BEGIN RSA PRIVATE KEY-----
```

❌ That is **NOT** a CSR
✔ That is the **key** — Name.com will reject it

---

### 3️⃣ Extra spaces / smart quotes / formatting damage

If you:

* Copied from WhatsApp
* Copied from Word / Notes
* Added spaces before `-----BEGIN`
* Changed line breaks

➡️ The CSR becomes invalid.

**Rule:** Copy directly from terminal → paste directly into Name.com.

---

### 4️⃣ Wrong Common Name (CN)

Name.com explicitly says:

> SSL will be installed on **[www.shishirrsiam.dev](http://www.shishirrsiam.dev)**

If your CSR was generated with:

```
Common Name = shishirrsiam.dev
```

instead of:

```
Common Name = www.shishirrsiam.dev
```

❌ It will fail validation.

---

### 5️⃣ You generated the CSR on Windows using online tools

Online CSR generators often:

* Use incorrect encoding
* Inject invisible characters

Name.com is strict → rejects them.

---

## Let’s VERIFY your CSR locally (no guessing)

On the server where you generated it, run:

```bash
openssl req -in shishirrsiam.dev.csr -noout -text
```

### ✅ If CSR is valid → you’ll see readable text

### ❌ If invalid → OpenSSL itself will error

If this command fails, **Name.com is correct**.
